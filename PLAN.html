project-agi working title

An open-source, self-hosted, configuration-driven Agent Intelligence stack — vertical-agnostic. SDK is the product; reference runtime + UI + chart are the supported distribution.

Works for any vertical with an OpenAPI surface and a need for agent tooling — telco, banking, retail, healthcare, energy, fleet, education, public sector, and beyond. v3 incorporates the resolutions from DEBATE.md: two-band architecture, pack-group-per-pod with claims-based dispatch, YAML-only prompts with a documented hotfix lane, Keycloak as the reference identity adapter, LangGraph + Pydantic-AI as blessed-but-optional orchestrators, plain async as default.

What a use case looks like

import litellm
from agi_sdk import use_case, serve

@use_case("bill_explainer", version="0.3.0")
class BillExplainer:
    def __init__(self, sdk):
        self.sdk       = sdk
        self.reasoning = sdk.models.binding("reasoning")
        self.billing   = sdk.mcp.tool("bluemarble.billing.adjust_charge")

    async def handle(self, request, ctx):                # plain async — the default
        prompt   = self.sdk.prompts.get("explain_bill").render(...)
        response = await litellm.acompletion(
            messages=[{"role": "user", "content": prompt}],
            **self.reasoning.kwargs(),
        )
        return response

if __name__ == "__main__":
    serve(BillExplainer, http=True, mcp=True)

§ 3Roles, not providers single biggest decoupling

Use case code never names a provider. It asks for a role. Operators bind roles → concrete models in their own config. Lint enforces: import openai / import anthropic / import boto3 requires an explicit waiver comment.

self.reasoning = sdk.models.binding("reasoning")   # ✓
self.fast      = sdk.models.binding("fast")        # ✓
self.claude    = sdk.models.binding("claude")      # ✓ (arbitrary string)
# operator.yaml
models:
  reasoning:
    model_id: bedrock/anthropic.claude-3-7-sonnet-20250219-v1:0
    region: eu-west-1
    default_params:
      max_tokens: 4096
      temperature: 0.3
  fast:
    model_id: bedrock/anthropic.claude-3-5-haiku-20241022-v1:0
    region: eu-west-1

Generalised from the v0 framework's mcpfyer. Points at any OpenAPI spec → emits a runnable MCP server. TMF is one example, not a dependency. Build-time generator — does not live in core runtime.

§ 5Orchestration — built-in + blessed-but-optional v2 · native built

Plain async def is the default. A built-in native orchestrator lives in-tree. Two adapters bless one external path each.

ChoiceWhen to pick itLicense / cost
Plain async def + Pydantic stateDefault. No framework gravity. Single-turn agents, request-scoped flows, fast prototypes.n/a · zero
sdk.orchestrators.native NEWMulti-step tool loops, HITL pause/resume, checkpointing, streaming — without LC types or a heavyweight framework.Apache-2.0 · in-tree · ~625 LOC
sdk.orchestrators.langgraphFull graph state-machine, subgraphs, the LangChain ecosystem.MIT (lib). Elastic-2.0 only at LangGraph Platform line, which we do not adopt.
sdk.orchestrators.pydantic_aiType-first agents, FastAPI-style ergonomics, Pydantic-native state without LangChain message types.MIT

External adapters stay thin (~150 LOC each):

  • Set OpenLLMetry baggage (bm.pack, bm.use_case, bm.tenant_id) on every span.
  • Resolve MCP tools from the pack's allow-list.
  • Do not wrap orchestration. Use-case authors talk to LangGraph or Pydantic-AI directly.

The native orchestrator is owned in-tree and provides the same surface (set_run_baggage, resolve_tools) plus a real implementation: Run, Orchestrator.step(), run_until_done(), pause/resume, pluggable CheckpointStore (Memory, FileJsonl, Mongo/Postgres stubs), and a streaming surface.

Why three options now (not two)

v1 of the research said "don't build" — LangGraph + Pydantic-AI compose cleanly enough. v2 (this revision) overrode that: building hedges watch-condition W-01 (LangGraph commercial-moat tightening) without requiring a port, gives users an Apache-2.0/no-LC-types path, and the real LOC came in at ~625 — manageable. Plain async stays the default. Care-intelligence still retrofits onto the langgraph adapter (no port — it already uses LangGraph in prod).

Watch conditions (when this flips)

  • LangChain Inc. tightens the commercial moat. If checkpointing/HITL move behind LangGraph Platform's Elastic-licensed runtime, demote LangGraph to "supported," promote Pydantic-AI or Strands Agents to blessed.
  • Use-case set narrows to durable workflows. If ≥80% of packs become long-running with retries+HITL, evaluate Temporal / Inngest / Hatchet outside the LLM-framework class. Plain async stays at LLM call sites; durability gets externalised.

Full research with 10-option survey, build-vs-adopt analysis, the v2 override, and per-vertical examples lives in ORCHESTRATOR_RESEARCH.html.

§ 6Multi-tenancy — pack-group-per-pod with claims-based dispatch

The deployment unit is one runtime container per tenant, hosting that tenant's packs, dispatched by X-Pack header. The framework's "one container per use case" rule is doctrinally correct but operationally fatal at telco scale (4 packs × 5 use cases × 2 replicas = 40 pods per customer).

Hardening mode

Single-pack-per-pod is supported as a Helm values flag for regulated tenants who buy isolation at the cost of baseline infra. Same chart, one toggle.

The leak the Framework Advocate flagged — closed by claims-based dispatch

  • X-Pack header is mandatory; missing → 400. No default-pack fallback.
  • X-Pack is validated against the authenticated tenant claim on every request. Mismatch → 401, not 200-with-wrong-data.
  • SDK contains no pack-dispatch logic. SDK takes one pack at a time. Header dispatch is purely an agi-runtime concern.
  • Storage queries are scoped on pack_slug. Fuzz test in CI verifies isolation.

§ 7Prompts — YAML-in-pack with a 15-minute hotfix lane

Prompts live as YAMLs in the pack (packs/<slug>/prompts/), baked into the runtime container at build, PR-reviewed. There is no DB-stored prompt with runtime editing. The admin UI shows prompts read-only.

Why not DB-stored, runtime-editable

The regulator-asks-why-did-the-refund-threshold-change-on-March-14 failure is severe. DB-stored runtime-editable prompts make this failure trivially easy to land — Ops hot-edits at 2am, no diff in source control, audit log captures who but not whether anyone reviewed it. YAMLs-in-repo + PR review is the only path that survives a telco audit.

The hotfix lane — ships with the product

  • Each pack is its own git repo (or a subdir of the customer's pack repo).
  • Convention: pack-hotfix/<ticket-id> branch.
  • Pre-merge automated smoke: KB load + prompt-render + tool-schema validation.
  • Auto-tag on merge → container rebuild + image push → runtime auto-reload on new image tag.
  • Promise: ≤ 15 minutes from PR merge to live.

What stays DB-backed

KB articles still get the version+restore mechanism (VersionType.MINOR) — facts updating is a different failure mode than behaviour changing. Prompts do not.

§ 9Configuration model

A pack is a folder. Loading is a function call. Switching the active pack is a header (X-Pack) — validated against the authenticated tenant claim.

packs/telco-demo/
├── pack.yaml          # identity, theme, role bindings, tool allow-list
├── tools.yaml         # optional pack-private tool declarations
├── prompts/           # YAML/j2, baked into container, source of truth
├── kb/                # markdown / JSON KB seeds
├── scenarios/         # optional demo scenarios (drives e2e)
└── assets/            # logos, favicons
slug: telco-demo
display_name: Telco Demo
vertical: telco
theme:
  primary: "#0066CC"
  mode_default: light
models:
  reasoning:
    model_id: openai/gpt-4o
    default_params: { temperature: 0.3 }
  fast:
    model_id: ollama/llama3.2
tools:
  allow:
    - billing.adjust_charge
    - kb.search
    - escalate_to_tier2
  deny: []
auth:
  required_scopes: ["agent:chat"]
  tenant_claim: "tenant_id"     # how X-Pack is validated against the token
trail:
  audit_sink: file-jsonl
observability:
  langfuse_endpoint: http://langfuse:3000

Config precedence (highest wins)

  1. Per-request headers (X-Pack — claims-validated, X-LLM-Override — admin only)
  2. Env vars (AGI_LLM_DEFAULT, AGI_STORAGE_URL, AGI_AUTH_MODE)
  3. Active pack's pack.yaml
  4. Operator-level operator.yaml (mounted via ConfigMap in k8s)
  5. Built-in defaults (SQLite, dev-noop, fake LLM in tests)

§ 12Resolved decisions from DEBATE.md

R1
Deployment unit = pack-group-per-pod with claims-based dispatch. Hardening mode (single-pack-per-pod) as a Helm values flag. SDK is pack-dispatch-agnostic; runtime owns header dispatch.
R2
Runtime + UI = first-class Band 2 packages. Same repo, distinct altitude from SDK/Core. Library is the heart, deployable is supported.
R3
Prompts = YAML-in-pack source of truth. 15-minute hotfix lane via pack-hotfix/<ticket> branch + auto-rebuild. No DB-stored runtime editing.
R4
Model runner = LiteLLM library mode. No custom abstraction.
R5
Telemetry = OpenLLMetry auto-instrument (only). AI-Trail wired off the OTel collector as audit-only sink.
R6
Identity = Keycloak first-class reference adapter. Generic OIDC interface. Three-role RBAC at dispatch layer.
R7
Auto-MCP = first-class agi-mcpfyer repo. Build-time generator, not buried in SDK utilities.
R8
Orchestration = blessed-but-optional langgraph + pydantic_ai adapters. Plain async is default.

§ 15Definition of done (v1.0)

  • pip install agi-sdk works on a clean machine — no Nexus, no VPN.
  • docker compose up launches the full Band-2 stack (runtime + Langfuse + Qdrant) in < 5 minutes.
  • A new tenant is added by dropping a folder under packs/ and restarting — zero code changes.
  • Auto-MCP hub takes any OpenAPI spec and produces a runnable MCP server. TMF is a reference example.
  • OpenLLMetry traces land in Langfuse with bm.pack / bm.use_case / bm.tenant_id overlaid.
  • Pack hotfix completes ≤ 15 min from PR merge to live.
  • Multi-pack-per-pod runtime rejects any request whose X-Pack doesn't match its tenant claim.
  • Care-intelligence runs on agi-sdk with the langgraph adapter in dev.
  • Isolation-gate, lint, type-check, test, image-build, helm-chart-against-kind all pass on main.
